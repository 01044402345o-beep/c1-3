# Make vs Zapier 비교분석 보고서

> **프로젝트**: 코딩 기초 퀴즈 자동화 시스템 (시나리오 A)
> **작성일**: 2025년
> **구현 내용**: Google Forms 퀴즈 제출 → 점수 판별 → 합격/불합격 시트 분리 기록 → 합격자 이메일 발송

---

## 1. 프로젝트 개요

### 구현 목표
```
Google Forms 퀴즈 제출
        ↓
점수 자동 판별 (80점 이상 / 미만)
        ↓
합격자 → 합격자 시트 기록 + 이메일 발송
불합격자 → 불합격자 시트 기록
```

### 사용 도구
| 도구 | 역할 |
|------|------|
| Google Forms | 퀴즈 10문항 제출 |
| Google Sheets | 합격자 / 불합격자 데이터 분리 저장 |
| Gmail | 합격자 자동 이메일 발송 |
| **Make** | 위 전체 자동화 구현 (방법 1) |
| **Zapier** | 위 전체 자동화 구현 (방법 2) |

---

## 2. 구현 구조 비교

### ✅ Make 구조 (1개 시나리오)
```
[Google Forms - Watch Responses]
            ↓
         [Router]
        ↙        ↘
  합격 경로      불합격 경로
  (80점 이상)    (80점 미만)
      ↓               ↓
[합격자 시트]   [불합격자 시트]
      ↓
[Gmail 발송]
```
> 💡 **1개의 시나리오**로 전체 흐름 완성

---

### ✅ Zapier 구조 (2개 Zap)
```
[Zap 1 - 합격자용]             [Zap 2 - 불합격자용]
Google Forms Trigger           Google Forms Trigger
        ↓                               ↓
Filter (80점 이상)             Filter (80점 미만)
        ↓                               ↓
Google Sheets (합격자 시트)    Google Sheets (불합격자 시트)
```
> ⚠️ **2개의 Zap**으로 분리 구현 필요 (무료 플랜 2단계 제약)

---

## 3. 항목별 비교분석

### 3-1. 무료 플랜 제약

| 항목 | Make | Zapier |
|------|------|--------|
| 무료 단계 수 | **제한 없음** | **2단계만 가능** |
| 시나리오/Zap 수 | 1개로 완성 | 2개 필요 |
| 월 실행 횟수 | 1,000회 | 100회 |
| 결론 | ✅ 유리 | ❌ 불리 |

> **분석**: Zapier 무료 플랜은 Trigger → Action의 2단계만 지원하여,
> Filter(조건 분기)를 별도 단계로 사용하면 Action을 추가할 수 없음.
> 이로 인해 합격/불합격 처리를 위해 Zap을 2개로 분리해야 하는 비효율이 발생함.
> Make는 이러한 단계 제한이 없어 1개의 시나리오로 모든 흐름을 완성할 수 있음.

---

### 3-2. 설정 난이도

| 항목 | Make | Zapier |
|------|------|--------|
| 전체 흐름 파악 | ✅ 시각적 캔버스로 한눈에 파악 | ❌ 목록형 구조로 흐름 파악 어려움 |
| 조건 분기 설정 | ✅ Router 모듈로 직관적 설정 | ❌ Filter Zap 별도 생성 필요 |
| 변수 연결 방식 | ✅ 클릭으로 자동 삽입 | ✅ 클릭으로 자동 삽입 |
| 모듈 연결 방식 | ✅ 드래그로 직관적 연결 | ❌ 순서대로 단계 추가 방식 |
| 결론 | ✅ 더 직관적 | ❌ 상대적으로 복잡 |

> **분석**: Make는 캔버스 기반의 시각적 편집 환경을 제공하여
> 전체 자동화 흐름을 한눈에 파악하고 수정할 수 있음.
> Zapier는 단계를 위에서 아래로 쌓는 목록형 구조로,
> 분기 처리 시 별도 Zap을 만들어야 해서 관리가 번거로움.

---

### 3-3. 조건 분기 처리

| 항목 | Make | Zapier |
|------|------|--------|
| 분기 방식 | Router 모듈 1개 | Zap 2개 분리 |
| 조건 설정 위치 | Router 내부 경로별 설정 | 각 Zap에 Filter 개별 설정 |
| 관리 편의성 | ✅ 1곳에서 통합 관리 | ❌ 2곳에서 각각 관리 |
| 수정 시 | ✅ 시나리오 1개만 수정 | ❌ Zap 2개 각각 수정 필요 |
| 결론 | ✅ 압도적 유리 | ❌ 불리 |

> **분석**: Make의 Router는 하나의 모듈 안에서 여러 경로를 설정할 수 있어
> 조건 추가/수정이 매우 간편함.
> Zapier는 조건별로 Zap을 새로 만들어야 하므로,
> 조건이 늘어날수록 관리해야 할 Zap 수도 함께 증가하는 구조적 한계가 있음.

---

### 3-4. 이메일 발송 기능

| 항목 | Make | Zapier |
|------|------|--------|
| 무료 플랜 구현 | ✅ 가능 (같은 시나리오 내 추가) | ❌ 무료 플랜 2단계 초과로 불가 |
| 설정 방법 | 합격 경로에 Gmail 모듈 추가 | 유료 플랜 필요 |
| 변수 연결 | ✅ Forms 데이터 직접 연결 | - |
| 결론 | ✅ 무료로 완성 | ❌ 유료 전환 필요 |

> **분석**: 이메일 발송은 시나리오 A의 핵심 기능 중 하나임.
> Make는 무료 플랜에서 단계 제한이 없으므로
> 합격 경로에 Gmail 모듈을 추가하는 것만으로 구현 완료됨.
> Zapier는 무료 플랜에서 이미 2단계(Trigger + Action)를 사용하고 있어
> 이메일 발송 Action을 추가하려면 유료 플랜으로 업그레이드해야 함.

---

### 3-5. 확장성

| 항목 | Make | Zapier |
|------|------|--------|
| 조건 추가 시 | ✅ Router에 경로만 추가 | ❌ Zap을 새로 생성해야 함 |
| 기능 추가 시 | ✅ 모듈 연결만 추가 | ❌ 단계 초과 시 유료 전환 필요 |
| 복잡한 자동화 | ✅ 1개 시나리오로 처리 가능 | ❌ 여러 Zap으로 분산 관리 |
| 결론 | ✅ 유리 | ❌ 불리 |

> **분석**: 자동화 시스템이 복잡해질수록 Make의 장점이 더욱 두드러짐.
> Make는 하나의 시나리오 안에서 모듈을 추가하는 방식으로 확장이 용이하지만,
> Zapier는 조건이나 기능이 늘어날 때마다 Zap을 추가로 생성해야 하며
> 무료 플랜의 단계 제한으로 인해 유료 전환 압박이 발생함.

---

## 4. 종합 비교표

| 비교 항목 | Make | Zapier |
|-----------|------|--------|
| 무료 플랜 단계 수 | ✅ 제한 없음 | ❌ 2단계 제한 |
| 구현 시나리오/Zap 수 | ✅ 1개 | ❌ 2개 이상 |
| 설정 난이도 | ✅ 직관적 (캔버스형) | ❌ 상대적으로 복잡 (목록형) |
| 조건 분기 처리 | ✅ Router로 통합 | ❌ Zap 분리 필요 |
| 이메일 발송 (무료) | ✅ 가능 | ❌ 불가 (유료 필요) |
| 확장성 | ✅ 높음 | ❌ 낮음 |
| **종합 평가** | ✅ **추천** | ❌ **비추천 (무료 기준)** |

---

## 5. 결론

> 이번 시나리오 A(코딩 기초 퀴즈 자동화)를 Make와 Zapier로 각각 구현한 결과,
> **무료 플랜 기준으로는 Make가 압도적으로 유리**하다는 결론을 도출하였다.

### Make를 추천하는 이유

1. **통합 구현 가능**: 1개의 시나리오로 Trigger, 조건 분기, 시트 기록, 이메일 발송까지 모두 처리
2. **직관적인 설정**: 캔버스 기반 시각적 편집으로 전체 흐름을 한눈에 파악 및 수정 가능
3. **무료 플랜 이메일 발송**: Zapier와 달리 단계 제한 없이 Gmail 발송까지 무료로 구현
4. **높은 확장성**: 조건이나 기능 추가 시 모듈만 연결하면 되어 유지보수가 간편
5. **관리 효율성**: 하나의 시나리오에서 모든 흐름을 통합 관리

### Zapier의 한계

1. **2단계 제한**: 무료 플랜에서 Trigger + Action만 가능하여 조건 분기 시 Zap 분리 필요
2. **이메일 발송 불가**: 무료 플랜에서 단계 초과로 이메일 발송 Action 추가 불가
3. **관리 복잡성**: 조건별로 Zap을 따로 만들어야 하여 수정 시 여러 곳을 변경해야 함
